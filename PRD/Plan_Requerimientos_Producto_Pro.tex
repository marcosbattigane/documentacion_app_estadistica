\section{Descripción General del Producto}

\subsection{Planteamiento del Problema}
El análisis estadístico en el básquet formativo sufre de una brecha crítica entre la recolección de datos y la utilidad táctica. Los entrenadores operan bajo condiciones de alta presión donde la toma de decisiones basada en datos es limitada por herramientas fragmentadas.

\begin{longtable}{lp{5cm}p{5cm}l}
\caption{Matriz de Problemas Operativos} \\
\toprule
\rowcolor{headergray} \textbf{\#} & \textbf{Problema} & \textbf{Impacto Operativo} & \textbf{Línea Base} \\
\midrule
1 & Carga manual ineficiente. & Interferencia táctica durante el juego; pérdida de foco. & > 20 min/partido. \\
2 & Ceguera estadística. & Decisiones basadas en intuición, no en métricas avanzadas (PPP, Eff). & Análisis 0\% automatizado. \\
3 & Fragmentación de datos. & Imposibilidad de seguimiento histórico o comparativo de rivales. & Datos aislados en papel/Excel. \\
4 & Inestabilidad de red. & Apps web fallan en estadios sin conectividad. & Requiere 100\% online. \\
5 & Rigidez de plataformas. & El DT no puede integrar los datos de "Ges Deportivo" fácilmente. & Ingesta manual post-partido. \\
\bottomrule
\end{longtable}

\subsection{Visión del Producto}
Desarrollar una aplicación multiplataforma (PC y Mobile) de ejecución local que centralice el conocimiento deportivo en un motor estadístico profesional. El sistema permitirá una gestión integral \textbf{sin necesidad de internet}, con un flujo de trabajo optimizado mediante la carga de planillas de "Ges Deportivo".

\subsection{Métricas de Éxito}
\begin{itemize}
    \item \textbf{MTTR (Time to Report):} Tiempo desde la carga del archivo hasta el informe completo < 2 minutos.
    \item \textbf{Tasa de Automatización:} \% de estadísticas de partido generadas sin intervención manual > 95\%.
    \item \textbf{Disponibilidad Offline:} 100\% de las funcionalidades críticas operativas sin conexión.
    \item \textbf{North Star Metric:} Número de sesiones de análisis táctico realizadas por el DT por semana.
\end{itemize}

\section{Arquitectura de Datos (Basado en DER)}
El sistema utiliza una base de datos relacional local (\textbf{SQLite}) con el siguiente esquema normalizado:

\subsection{Entidades Principales}
\begin{itemize}
    \item \textbf{Gestión de Identidad:} \texttt{usuario}, \texttt{club}, \texttt{usuarioClub} (N:M).
    \item \textbf{Estructura Deportiva:} \texttt{jugador}, \texttt{categoria}, \texttt{competencia}, \texttt{inscripcion}.
    \item \textbf{Gestión de Listas:} \texttt{listaBuenaFe}, \texttt{jugadorListaBuenaFe}.
    \item \textbf{Eventos y Estadísticas:} \texttt{partido}, \texttt{jugadorPartido} (Contiene 20+ métricas: Puntos, T2, T3, TL, Rebotes, Asistencias, etc.).
\end{itemize}

\subsection{Reglas de Integridad}
\begin{itemize}
    \item \textbf{Vínculos:} Relación 1:1 entre \texttt{inscripcion} y \texttt{listaBuenaFe}.
    \item \textbf{Persistencia:} El historial de un jugador en diferentes clubes se mantiene vía \texttt{jugadorClub}.
    \item \textbf{Métricas:} La tabla \texttt{jugadorPartido} actúa como el hecho (fact table) para el motor de análisis.
\end{itemize}

\section{Arquitectura de Software (Clean + Hexagonal)}
Para cumplir portabilidad, testabilidad y ejecución local-first, el sistema adopta una Arquitectura Limpia con enfoque Hexagonal.

\subsection{Capas y Responsabilidades}
\begin{itemize}
    \item \textbf{Dominio (\texttt{src/domain}):} Entidades puras, reglas de negocio, excepciones e interfaces (ports) sin dependencias externas.
    \item \textbf{Aplicación (\texttt{src/application}):} Casos de uso y DTOs; orquesta reglas usando inyección de dependencias.
    \item \textbf{Infraestructura (\texttt{src/infrastructure}):} Repositorios SQLite, parser Excel (Pandas), UI CLI/GUI y generación de reportes.
\end{itemize}

\subsection{Regla de Dependencias}
\begin{itemize}
    \item \texttt{domain <- application <- infrastructure}
    \item \texttt{domain} no puede importar \texttt{application} ni \texttt{infrastructure}.
    \item \texttt{application} puede importar \texttt{domain}, pero no \texttt{infrastructure}.
\end{itemize}

\subsection{Patrones de Diseño}
\begin{itemize}
    \item \textbf{Repository Pattern:} Abstracción de persistencia por agregado (\texttt{UserRepository}, \texttt{PlayerRepository}, etc.).
    \item \textbf{Dependency Injection:} Los casos de uso reciben puertos, no implementaciones concretas.
    \item \textbf{Command Pattern:} CLI extensible por subcomandos sin bloques monolíticos \texttt{if/else}.
\end{itemize}

\subsection{Estructura Objetivo de Directorios}
\begin{itemize}
    \item \texttt{src/domain/entities, interfaces, exceptions}
    \item \texttt{src/application/use\_cases, dtos, services}
    \item \texttt{src/infrastructure/persistence, analytics, ingest, reports, ui}
    \item \texttt{tests/unit, tests/integration}
\end{itemize}

\section{Gestión de Tareas y Plan de Entregas (Consolidado)}

\subsection{Hito 1: Núcleo de Datos e Interfaz CLI (v0.1)}
\textbf{Objetivo del hito:} construir una base técnica funcional por CLI, con persistencia robusta, autenticación local, casos de uso operativos y validaciones de integridad.

\textbf{Épica E1 — Infraestructura y Persistencia}
\begin{itemize}
    \item \textbf{US-101: Implementación de Base de Datos SQLite y Vistas}
    \begin{itemize}
        \item \textit{Objetivo funcional:} habilitar un esquema relacional local verificable para todo el ciclo del producto.
        \item \textit{Capa de infraestructura:} clase \texttt{SQLiteManager} en \texttt{src/infrastructure/persistence/database\_manager.py} con métodos de conexión e inicialización atómica.
        \item \textit{Base SQL obligatoria:} \texttt{src/infrastructure/persistence/sql/schema.sql}, \texttt{views.sql}, \texttt{seed.sql}.
        \item \textit{Criterios de aceptación técnicos:}
        \begin{itemize}
            \item AC1: schema idempotente con \texttt{CREATE TABLE IF NOT EXISTS}, FKs activas, \texttt{ON DELETE/UPDATE CASCADE} y \texttt{CHECK} de métricas.
            \item AC2: vistas operativas \texttt{v\_partidos\_resumen}, \texttt{v\_boxscore\_completo}, \texttt{v\_jugador\_totales\_temporada}, \texttt{v\_listas\_detalle}.
            \item AC3: protección de división por cero mediante \texttt{NULLIF} o \texttt{CASE}.
            \item AC4: datos semilla útiles para validar consultas y motor estadístico.
        \end{itemize}
        \item \textit{Reglas de negocio DB:} \texttt{idClubLocal != idClubVisitante}, coherencia \texttt{fechaHasta >= fechaDesde}, unicidad en \texttt{listaBuenaFe.idInscripcion}.
        \item \textit{Testing mínimo:} \texttt{tests/integration/test\_database\_schema.py} cubriendo existencia de tablas/vistas, integridad referencial, constraints, seed y consultas de vistas.
    \end{itemize}

    \item \textbf{US-102: DatabaseManager y Patrón Repository}
    \begin{itemize}
        \item \textit{Objetivo funcional:} separar acceso a datos por agregados de dominio bajo contratos tipados.
        \item \textit{Dominio (ports):}
        \texttt{src/domain/interfaces/user\_repository.py},
        \texttt{club\_repository.py},
        \texttt{player\_repository.py},
        \texttt{game\_repository.py}.
        \item \textit{Infraestructura (implementaciones):}
        \texttt{src/infrastructure/persistence/sqlite\_user\_repository.py},
        \texttt{sqlite\_club\_repository.py},
        \texttt{sqlite\_player\_repository.py},
        \texttt{sqlite\_game\_repository.py}.
        \item \textit{Criterios de aceptación técnicos:}
        \begin{itemize}
            \item AC1: \texttt{connect()} activa FKs y \texttt{row\_factory=sqlite3.Row}.
            \item AC2: los repositorios retornan dataclasses, no tuplas SQL.
            \item AC3: la capa \texttt{domain} no importa \texttt{sqlite3/pandas}.
            \item AC4: guardado de partido + boxscore con transacción atómica.
        \end{itemize}
        \item \textit{Testing mínimo:} \texttt{tests/integration/test\_repositories.py} con CRUD, búsquedas, mapeos y atomicidad.
    \end{itemize}
\end{itemize}

\section{Definición de "Hecho" (Definition of Done)}
Una historia de usuario se considera \textbf{Hecha} cuando:
\begin{enumerate}
    \item El código está integrado en la rama principal sin conflictos.
    \item Pasa pruebas unitarias e integración con cobertura > 80\% en componentes no críticos.
    \item Los componentes críticos alcanzan cobertura > 95\% (incluyendo pero no limitado a autenticación, persistencia transaccional, rollback y fórmulas estadísticas).
    \item La clasificación crítica/no crítica se define y versiona en el catálogo técnico del backlog.
    \item Se respeta la arquitectura de capas (sin imports cruzados entre \texttt{domain} y \texttt{infrastructure}).
    \item Si hay cambios SQL, la vista/tabla está versionada y cubierta por test de integración con datos semilla.
    \item Si la US persiste múltiples tablas, existe evidencia de transacción atómica y rollback probado.
    \item El ADR correspondiente fue aprobado y documentado.
    \item La interfaz fue validada en al menos dos plataformas (ej. Desktop y Mobile).
    \item Métodos públicos relevantes incluyen docstring y trazabilidad mínima en logs de operación.
\end{enumerate}

\subsection{Catálogo técnico de criticidad}
El catálogo de criticidad es un artefacto vivo del backlog de arquitectura donde se lista cada módulo, su nivel de criticidad (crítico/no crítico), justificación, dueño técnico y evidencia de cobertura objetivo. Su mantenimiento se realiza en cada refinamiento de hito y es prerrequisito para validar el DoD.

\textbf{Formato y ubicación:} archivo versionado \texttt{ADR/catalogo-criticidad.md} con tabla mínima: \texttt{Modulo | Criticidad | Justificación | Owner | Cobertura objetivo | Evidencia}. El catálogo se inicializa en Hito 1 y se actualiza en cada planificación de sprint.

\section{Hoja de Ruta y Futuras Expansiones}
Para transformar StatsPro Basketball en una plataforma líder de inteligencia deportiva, se proponen las siguientes fases de expansión post-v1.0:

\subsection{Hito 5: Ecosistema Conectado y Colaborativo (v1.5)}
\textbf{Objetivo:} Romper el aislamiento local para permitir la colaboración entre cuerpos técnicos.
\begin{itemize}
    \item \textbf{Sincronización Cloud Híbrida:} Implementación de un backend (ej. FastAPI + PostgreSQL) para respaldar datos en la nube y permitir el uso multidispositivo (PC en oficina, Tablet en cancha).
    \item \textbf{Módulo de Exportación para Redes:} Generación de "Match Cards" automáticas con las figuras del partido para compartir directamente en Instagram/X del club.
    \item \textbf{Base de Datos Compartida de Scouting:} Repositorio comunitario donde DTs pueden compartir datos de rivales (bajo consentimiento), creando un mapa de scouting regional.
\end{itemize}

\subsection{Hito 6: Inteligencia Táctica Basada en IA (v2.0)}
\textbf{Objetivo:} Pasar del análisis descriptivo (¿qué pasó?) al análisis prescriptivo (¿qué debería pasar?).
\begin{itemize}
    \item \textbf{Motor de Recomendaciones Tácticas:} Uso de modelos de ML para sugerir quintetos iniciales basados en el "Match-up" contra el rival y el rendimiento reciente.
    \item \textbf{Detección Automática de Patrones:} Identificación de rachas de tiro o debilidades defensivas durante la carga de datos, alertando al DT sobre tendencias críticas.
    \item \textbf{Integración de Video Scouting:} Vinculación de eventos estadísticos con clips de video (ej. vía integración con YouTube o archivos locales) para sesiones de video-análisis rápidas.
\end{itemize}

\subsection{Hito 7: Live Stats y Scouting en Tiempo Real (v3.0)}
\textbf{Objetivo:} Capturar datos en el momento en que ocurren, eliminando la dependencia exclusiva de Excel externos.
\begin{itemize}
    \item \textbf{Módulo de Captura en Vivo:} Interfaz táctil optimizada para que un asistente cargue eventos (tiros, pérdidas, faltas) en tiempo real desde una tablet.
    \item \textbf{Live Dashboard para el Banco:} Visualización de métricas avanzadas (PPP, eFG%) en vivo para ajustes tácticos durante los tiempos muertos.
    \item \textbf{Control de Carga y Fatiga:} Integración con wearables o entrada manual de minutos para gestionar el esfuerzo físico de los jugadores jóvenes a lo largo de torneos intensos.
\end{itemize}
