\section{Descripción General del Producto}

\subsection{Planteamiento del Problema}
El análisis estadístico en el básquet formativo sufre de una brecha crítica entre la recolección de datos y la utilidad táctica. Los entrenadores operan bajo condiciones de alta presión donde la toma de decisiones basada en datos es limitada por herramientas fragmentadas.

\begin{longtable}{lp{5cm}p{5cm}l}
\caption{Matriz de Problemas Operativos} \\
\toprule
\rowcolor{headergray} \textbf{\#} & \textbf{Problema} & \textbf{Impacto Operativo} & \textbf{Línea Base} \\
\midrule
1 & Carga manual ineficiente. & Interferencia táctica durante el juego; pérdida de foco. & > 20 min/partido. \\
2 & Ceguera estadística. & Decisiones basadas en intuición, no en métricas avanzadas (PPP, Eff). & Análisis 0\% automatizado. \\
3 & Fragmentación de datos. & Imposibilidad de seguimiento histórico o comparativo de rivales. & Datos aislados en papel/Excel. \\
4 & Inestabilidad de red. & Apps web fallan en estadios sin conectividad. & Requiere 100\% online. \\
5 & Rigidez de plataformas. & El DT no puede integrar los datos de "Ges Deportivo" fácilmente. & Ingesta manual post-partido. \\
\bottomrule
\end{longtable}

\subsection{Visión del Producto}
Desarrollar una aplicación multiplataforma (PC y Mobile) de ejecución local que centralice el conocimiento deportivo en un motor estadístico profesional. El sistema permitirá una gestión integral \textbf{sin necesidad de internet}, con un flujo de trabajo optimizado mediante la carga de planillas de "Ges Deportivo".

\subsection{Métricas de Éxito}
\begin{itemize}
    \item \textbf{MTTR (Time to Report):} Tiempo desde la carga del archivo hasta el informe completo < 2 minutos.
    \item \textbf{Tasa de Automatización:} \% de estadísticas de partido generadas sin intervención manual > 95\%.
    \item \textbf{Disponibilidad Offline:} 100\% de las funcionalidades críticas operativas sin conexión.
    \item \textbf{North Star Metric:} Número de sesiones de análisis táctico realizadas por el DT por semana.
\end{itemize}

\section{Metas y No Metas}

\subsection{En el Alcance (v1.0)}
\begin{itemize}
    \item \textbf{Motor de Ingesta:} Subida manual de planillas Excel descargadas de "Ges Deportivo".
    \item \textbf{Persistencia Local:} Base de datos SQLite única por instalación de usuario.
    \item \textbf{Motor Estadístico:} Análisis de tendencias, PPP, porcentajes por zona y eficiencia con Pandas.
    \item \textbf{Multiplataforma:} Ejecución en Windows, Linux, Mac y dispositivos móviles.
\end{itemize}

\subsection{Fuera de Alcance}
\begin{itemize}
    \item \textbf{Sincronización Cloud:} No se contempla almacenamiento en la nube inicialmente.
    \item \textbf{Importación Automática/API:} No hay conexión directa con servidores de Ges Deportivo (flujo manual).
    \item \textbf{IA Predictiva:} No se incluirán modelos de predicción de resultados en v1.0.
\end{itemize}

\section{Arquitectura de Datos (Basado en DER)}
El sistema utiliza una base de datos relacional local (\textbf{SQLite}) con el siguiente esquema normalizado:

\subsection{Entidades Principales}
\begin{itemize}
    \item \textbf{Gestión de Identidad:} \texttt{usuario}, \texttt{club}, \texttt{usuarioClub} (N:M).
    \item \textbf{Estructura Deportiva:} \texttt{jugador}, \texttt{categoria}, \texttt{competencia}, \texttt{inscripcion}.
    \item \textbf{Gestión de Listas:} \texttt{listaBuenaFe}, \texttt{jugadorListaBuenaFe}.
    \item \textbf{Eventos y Estadísticas:} \texttt{partido}, \texttt{jugadorPartido} (Contiene 20+ métricas: Puntos, T2, T3, TL, Rebotes, Asistencias, etc.).
\end{itemize}

\subsection{Reglas de Integridad}
\begin{itemize}
    \item \textbf{Vínculos:} Relación 1:1 entre \texttt{inscripcion} y \texttt{listaBuenaFe}.
    \item \textbf{Persistencia:} El historial de un jugador en diferentes clubes se mantiene vía \texttt{jugadorClub}.
    \item \textbf{Métricas:} La tabla \texttt{jugadorPartido} actúa como el hecho (fact table) para el motor de análisis.
\end{itemize}

\section{Gestión de Tareas y Plan de Entregas}

\subsection{Hito 1: Cimentación y Modelo Relacional (v0.1)}
\textbf{Epic E1: Persistencia y Entidades}
\begin{itemize}
    \item \textbf{US-101: Implementación de la Base de Datos SQLite}
    \begin{itemize}
        \item \textit{Story:} Como desarrollador, quiero crear el esquema relacional completo para asegurar la integridad de los datos deportivos.
        \item \textit{Criterios de Aceptación:}
        \begin{itemize}
            \item AC1: Script de creación de tablas (\texttt{DDL}) validado contra el DER.
            \item AC2: Implementación de \texttt{Triggers} para cálculos automáticos simples (ej. puntos totales).
            \item AC3: Pruebas de carga con 1,000 registros de jugadores sin degradación de latencia.
            \item AC4: Generación de un Diccionario de Datos técnico documentado en el repositorio.
        \end{itemize}
    \end{itemize}
    \item \textbf{US-102: Módulo de Gestión de Clubes y Usuarios}
    \begin{itemize}
        \item \textit{Story:} Como DT, quiero registrar mi cuenta local y mis clubes para segmentar mis estadísticas.
        \item \textit{Criterios de Aceptación:}
        \begin{itemize}
            \item AC1: Pantalla de login/registro local con cifrado de contraseña (SHA-256).
            \item AC2: Soporte para múltiples clubes bajo un mismo perfil de usuario.
            \item AC3: Implementación de roles (\texttt{rolEntrenador}) que limiten la edición de datos críticos.
        \end{itemize}
    \end{itemize}
\end{itemize}

\subsection{Hito 2: Motor de Ingesta e Inteligencia de Datos (v0.2)}
\textbf{Epic E2: Integración "Ges Deportivo"}
\begin{itemize}
    \item \textbf{US-201: Parseo de Planillas Excel con Pandas}
    \begin{itemize}
        \item \textit{Story:} Como analista, quiero procesar los archivos de Ges Deportivo para eliminar el error humano en la transcripción.
        \item \textit{Criterios de Aceptación:}
        \begin{itemize}
            \item AC1: Mapeo exitoso de columnas de Excel a campos de la tabla \texttt{jugadorPartido}.
            \item AC2: Manejo de errores: Si un jugador no está en la base de datos, el sistema debe ofrecer "Crear Nuevo" o "Vincular Existente".
            \item AC3: Verificación de consistencia: La suma de puntos individuales debe coincidir con el resultado final del partido (margen de error 0).
            \item AC4: Log de importación que detalle registros procesados, saltados y errores encontrados.
        \end{itemize}
    \end{itemize}
\end{itemize}

\textbf{Epic E3: Motor Estadístico Pro}
\begin{itemize}

\section{Definición de "Hecho" (Definition of Done)}
Una historia de usuario se considera \textbf{Hecha} cuando:
\begin{enumerate}
    \item El código está integrado en la rama principal sin conflictos.
    \item Pasa las pruebas unitarias y de integración (Cobertura > 80\%).
    \item El ADR correspondiente ha sido aprobado y documentado.
    \item La interfaz ha sido validada en al menos dos plataformas (ej. Desktop y Mobile).
    \item No existen fugas de memoria ni degradación del rendimiento en SQLite.
\end{enumerate}
